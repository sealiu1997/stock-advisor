# 需求清单

下面是我建议交给上游 agent 的正式需求清单，分成 PKS upstream 和 stock-advisord / stock-advisor 两部分。

# A. PKS upstream 改动需求

## A1. WebUI 性能与可用性

### 背景

试运行中，`market-context` capsule 在经历大量 candidate / accepted / rejected audit claim 后，WebUI 出现：

- `/` dashboard 超时；
- `/projects/{project_id}/review` 超时；
- `pks serve` 进程 CPU 100%；
- API `/api/projects` 仍可用，说明是 HTML 页面同步重计算导致。

### 需求

#### 1. Dashboard 不应同步执行重型 `health_check`

当前：

代码· python

```
kernel.health_check(project.project_id)
```

在首页同步执行，导致页面阻塞。

建议：

- 首页只展示 capsule metadata + lightweight counters；
- health check 改成异步 API；
- 支持缓存最近一次 health report；
- WebUI 页面通过 htmx/fetch 拉取；
- 如果 health check 超时，页面仍可显示。

验收：

- 1000+ claims / 500+ audit claims 下，`GET /` 首屏 < 1s；
- health API 可单独调用，超时时不阻塞 dashboard。

#### 2. Review 页面分页 / lazy review

当前 Review 页面会：

代码· python

```
for every candidate:
    kernel.review_candidate(...)
```

建议：

- `GET /review` 默认只显示第一页，例如 25 条；
- review decision 按单条懒加载；
- 支持 query params：
  - `?limit=25`
  - `?cursor=...`
  - `?tag=...`
  - `?predicate=...`
  - `?type=...`
  - `?source=...`
- batch accept/reject 只对当前选择 ID 操作。

验收：

- 1000 candidates 下 Review 页 < 1s 首屏；
- 单条 review decision API < 500ms；
- batch accept/reject 返回 per-ID result。

#### 3. 增加 WebUI 绑定 host / port 配置

当前需要手写 LaunchAgent：

代码· bash

```
pks serve --host 0.0.0.0 --port 8420
```

建议：

- `pks serve` 支持配置文件；
- 文档明确：
  - localhost only；
  - LAN；
  - auth/reverse proxy；
- WebUI 默认仍应 localhost，但可显式配置 LAN。

验收：

代码· bash

```
pks serve --host 0.0.0.0 --port 8420
```

可被局域网访问。

## A2. Claim 生命周期模型增强

### 背景

当前 `Claim` 有：

代码· python

```
valid_until: date | None
```

但 CLI/API/helper 写入路径不方便设置，projection/health 对短周期市场数据支持不足。

### 需求

#### 1. Candidate creation 支持 `valid_until`

所有入口需要支持：

- Python API；
- CLI；
- MCP；
- WebUI form；
- JSON API。

验收：

代码· bash

```
pks claim submit ... --valid-until 2026-06-23
```

或等价 API 可创建带过期时间的 candidate。

#### 2. 增加 TTL / lifecycle policy override

市场类 fact 需要短生命周期：

- intraday price signal：当天或次日过期；
- scheduled calendar event：公布后 1 天过期；
- actual macro data release：7-30 天；
- durable macro regime fact：30-90 天；
- inference narrative：7-30 天。

建议支持：

代码· yaml

```
lifecycle:
  factual:
    stale_after_days: 90
  by_tag:
    intraday:
      valid_for_hours: 24
    scheduled_event:
      valid_for_days: 1
    actual_release:
      valid_for_days: 30
    narrative:
      stale_after_days: 14
```

验收：

- `pks maintenance` 能自动 expire/stale tag-specific claims；
- projection 默认排除 expired。

#### 3. 区分 `stale` 与 `expired`

市场数据里：

- `stale` = 需要重新确认，但仍可作历史；
- `expired` = 不应进入当前上下文。

验收：

- projection 可配置：
  - include accepted but stale；
  - exclude expired；
  - include historical archive。

## A3. 支持 market domain 或 domain 扩展

### 背景

当前 `ProjectMetadata.domain` 是 enum：

代码· python

```
CapsuleDomain
```

不能只靠 YAML 新增：

代码· text

```
domain: market
```

### 需求

两种可选方案：

#### 方案 1：让 domain 可扩展

允许 domain 为字符串，不限制 enum。

#### 方案 2：内置 `market` domain

新增：

代码· python

```
CapsuleDomain.MARKET = "market"
```

并提供默认 policy。

建议默认：

domain: market lifecycle:  factual:    stale_after_days: 30    auto_accept_threshold: 0.95  inference:    stale_after_days: 14 manual_review_types:  - inference  - preference  - constraint

验收：

代码· bash

```
pks new market-context --domain market ...
pks policy validate market
```

可用。

## A4. 支持结构化 claim metadata

### 背景

现在 calendar 事件被塞进字符串：

代码· text

```
"美国5月核心PCE物价指数年率 on 2026-06-25 20:30"
```

前值、预期、实际值也塞进 excerpt：

代码· text

```
前值: ..., 预期: ..., 实际: ...
```

这不适合长期查询和 projection。

### 需求

给 Claim 增加结构化 metadata 字段，例如：

代码· python

```
metadata: dict[str, Any] = {}
```

或者至少允许 `qualifier` / `content` 保存 JSON 并在 WebUI 中友好展示。

Calendar 事件应支持：

代码· json

```
{
  "event_name": "US Core PCE",
  "release_time": "2026-06-25T20:30:00+08:00",
  "period": "2026-05",
  "previous": "...",
  "consensus": "...",
  "actual": null,
  "status": "scheduled"
}
```

Actual release：

代码· json

```
{
  "event_name": "US Core PCE",
  "period": "2026-05",
  "previous": "...",
  "consensus": "...",
  "actual": "...",
  "surprise": "hotter_than_expected"
}
```

验收：

- WebUI 可以显示 metadata；
- projection templates 可以引用 metadata；
- API 返回 metadata。

## A5. 支持 transient / scratchpad material，不进入 durable PKS

### 背景

我们的试运行证明：大量新闻只适合作为“当日播报临时材料”，不应该进入 durable candidate queue。

### 需求

PKS 可以新增一层非 durable material：

代码· text

```
Scratchpad / Evidence Inbox / Daily Material
```

特性：

- 可按 project/date 存储；
- 默认 TTL；
- 不进入 accepted/candidate claim；
- 可作为生成 candidate 的 evidence；
- 可被每日 agent 读取；
- 支持自动清理。

示例：

代码· bash

```
pks material add market-context --source jin10 --date 2026-06-22 --json ...
pks material list market-context --date today --tag macro
pks material promote market-context <material_id> --as factual
```

验收：

- raw news 可以存 material；
- selected facts 才 promote 成 candidate；
- material 自动按 TTL 清理。

## A6. Candidate queue 需要治理能力

需求：

- candidate age filter；
- source filter；
- batch reject by query；
- exact duplicate detection；
- near duplicate detection；
- group by subject/predicate/source/tag；
- daily counts。

CLI 示例：

代码· bash

```
pks review list market-context --tag news --limit 50
pks review reject-query market-context --tag news --older-than 1d
pks review dedupe market-context --predicate headline --keep latest
```

验收：

- 能清理重复候选而不用写外部脚本；
- WebUI 和 CLI 一致。

# B. stock-advisord / stock-advisor 改动需求

## B1. 核心架构调整：raw material 与 durable PKS claim 分离

### 背景

当前 watcher 把大量 Jin10/RSS/price/calendar 信息直接写成 PKS candidate，导致：

- candidate 队列膨胀；
- Review 页不可用；
- 垃圾 claim 需要人工/脚本清理；
- durable knowledge 与当日临时新闻混杂。

### 需求

改成两层：

text Daily Raw Material Layer

- Jin10 flash
- Jin10 calendar
- RSS
- price snapshots
- yfinance/news
- 当日临时分析材料
- 每日自动刷新/过期

Durable PKS Claim Layer

- 每日最多 <=20 factual claims
- 每日最多 <=5 inference claims
- 有 valid_until
- 有 evidence
- 可进入 projection / future briefing

目录建议：

text data/daily_materials/YYYY-MM-DD/ raw_jin10.jsonl raw_rss.jsonl raw_calendar.jsonl raw_prices.jsonl selected_facts.json selected_inferences.json briefing_context.md

验收：

- raw news 默认不进入 PKS candidate；
- 每天自动新建材料目录；
- 旧材料可按 3-7 天清理或归档。

## B2. 每日 durable claim 数量上限

用户要求：

代码· text

```
daily factual claims <= 20
daily inference claims <= 5
```

### 需求

新增 daily selector：

代码· text

```
material -> scoring -> top facts -> top inferences -> PKS candidate
```

选择逻辑考虑：

- 持仓；
- watchlist；
- US/HK 市场；
- 黄金；
- BTC；
- FOMC / CPI / PCE / NFP / 美债 / DXY；
- geopolitical / oil / tariffs；
- price reaction；
- source credibility；
- 是否影响今日/明日播报主线。

验收：

- 单日进入 PKS 的 factual candidate 不超过 20；
- 单日进入 PKS 的 inference candidate 不超过 5；
- 超出部分只留在 daily material；
- 生成一份 selection report。

示例：

代码· json

```
{
  "date": "2026-06-22",
  "factual_selected": 18,
  "inference_selected": 4,
  "discarded_to_scratchpad": 312,
  "reasons": [...]
}
```

### 当前问题

现在是：

代码· text

```
predicate: upcoming_event
object: "{event_name} on {event_date}"
excerpt: "前值: ..., 预期: ..., 实际: ..."
```

### 需求

#### Scheduled event

代码· json

```
{
  "type": "factual",
  "subject": "economic_calendar",
  "predicate": "scheduled_event",
  "metadata": {
    "event_name": "...",
    "release_time": "...",
    "previous": "...",
    "consensus": "...",
    "actual": null,
    "status": "scheduled"
  },
  "tags": ["calendar", "scheduled", "macro"],
  "valid_until": "release_date + 1 day"
}
```

#### Actual release

代码· json

```
{
  "type": "factual",
  "subject": "macro_release",
  "predicate": "actual_release",
  "metadata": {
    "event_name": "...",
    "period": "...",
    "previous": "...",
    "consensus": "...",
    "actual": "...",
    "surprise": "hotter_than_expected"
  },

```

验收：

- calendar scheduled event 自动过期；
- actual release 单独写入；
- previous / consensus / actual 可被 projection 读取。

## B4. Price signal 类型修正

### 当前问题

现在 overview 写入：

text type: inference predicate: index_move / commodity_move object: 黄金 4177.8 (-1.1%)

这是错的。

### 需求

价格/指数/商品移动应为 factual：

代码· json

```
{
  "type": "factual",
  "subject": "gold",
  "predicate": "market_move",
  "metadata": {
    "asset": "gold",
    "price": 4177.8,
    "change_pct": -1.1,
    "observed_at": "...",
    "window": "1d"
  },
  "tags": ["price", "intraday", "commodity"],
  "valid_until": "today + 1 day"
}
```

真正 inference 应该是 agent 生成：

代码· text

```
黄金走弱可能反映中东风险缓和 + 利率预期上行，而非单纯技术调整。
```

验收：

- `write_market_signal()` 不再用于 raw price move；
- `write_price_fact()` 或 `write_market_move()` 负责 factual price signal；
- price fact 默认短 TTL。

## B5. Inference 必须由 agent 筛选，不由 keyword analyzer 自动大量生成

### 当前问题

`analyzer.py` 通过关键词自动生成 active_theme，噪音很大。

### 需求

新的 inference selector 应该：

1. 读取当日 raw material；
2. 读取 holdings/watchlist；
3. 读取 market moves；
4. 判断主线；
5. 最多生成 5 条 inference candidate；
6. 每条 inference 必须引用 2-5 条 evidence/fact/material；
7. 有 confidence；
8. 有 valid_until；
9. 有影响范围：
   - US market；
   - HK market；
   - gold；
   - BTC；
   - holdings；
   - watchlist。

验收：

- 单日 inference candidate <= 5；
- 无 source evidence 的 inference 不允许写入 PKS；
- keyword analyzer 只能做初筛，不直接写 PKS。

## B6. News 默认不进 PKS

### 当前问题

Jin10/RSS headline 大量进入 PKS candidate。

### 需求

新闻处理分层：

text raw headline -> daily material important event -> selected fact candidate narrative synthesis -> selected inference candidate

默认：

- raw headline 不进 PKS；
- RSS 新闻不进 PKS；
- Jin10 flash 不进 PKS；
- 只有 daily selector 选中后才进 PKS。

验收：

- 一天 500 条 raw news 不会导致 500 candidate；
- candidate queue 可控；
- rejected audit 不再膨胀。

## B7. Deduplication 必须稳定

### 已发现问题

之前用 Python：

代码· python

```
hash(str(item))
```

这在不同进程间不稳定，导致重启后重复入库。

### 需求

所有 source item 使用稳定 ID：

代码· python

```
sha256(normalized_title + source + published_at)
```

或 source 原始 ID。

用于：

- Jin10 flash；
- RSS；
- calendar；
- overview signal；
- price moves。

验收：

- watcher 重启不会重复写同一新闻；
- state 文件可 seed；
- exact duplicate / near duplicate 可统计。

## B8. Daily cleanup job

### 需求

每天自动：

- 清理昨日 raw material；
- expire yesterday intraday price facts；
- expire past scheduled events；
- refresh projections；
- 输出 cleanup report。

示例：

代码· json

```
{
  "expired_intraday_facts": 12,
  "expired_scheduled_events": 8,
  "raw_material_archived": 420,
  "candidate_queue_remaining": 23
}
```

## B9. Projection 改成消费 selected durable claims

日报技能应优先读：

text selected_facts selected_inferences accepted durable claims

而不是扫全部新闻/候选。

建议 projection：

text briefing_context.md daily_selected_facts.md daily_selected_inferences.md macro_calendar.md market_moves.md

验收：

- 每日播报上下文稳定、短小；
- 不超过约 20 facts + 5 inferences；
- projection 中不出现 raw headline 垃圾。

# C. 本次试运行暴露的关键问题摘要

可以直接给上游 agent：

1. PKS WebUI 对大 capsule 不可用 Dashboard / Review 同步重计算导致超时和 100% CPU。
2. candidate queue 被当作 raw news inbox 使用 这是架构错误。candidate 应该是“待审核知识”，不是“所有抓到的材料”。
3. news/material 与 durable knowledge 混层 Jin10/RSS/price/calendar 原始材料应该先进入 daily scratchpad。
4. price move 被错误建模为 inference `黄金 -1.1%` 是 factual market move，不是 inference。
5. calendar 没有生命周期与结构化字段 scheduled / expected / actual / surprise / valid_until 需要分离。
6. 缺少数量上限 用户目标是每天 durable fact <=20，inference <=5。
7. inference 应由 agent 根据重要性、持仓、关注列表筛选 不应由关键词 analyzer 自动大量生成。
8. 需要真正的 market domain 或可扩展 domain 当前只能复用 research policy，不够干净。

## 我建议你给上游 agent 的一句总需求

> 请把 PKS + stock-advisord 改造成“两层知识治理架构”：raw market materials 进入每日临时 scratchpad，只有经过 agent 基于持仓/关注列表/宏观重要性筛选后的少量 facts 和 inferences 才进入 PKS durable claims；同时修复 PKS WebUI 在大 capsule 下的 dashboard/review 性能问题，并支持 market domain、per-claim TTL、结构化 calendar/price metadata。
